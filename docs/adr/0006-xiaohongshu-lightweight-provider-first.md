# Xiaohongshu Lightweight Provider First

Xiaohongshu support keeps a lightweight provider boundary for metadata/media/transcription while treating `MediaCrawler` as the managed external runtime for comments. The main project depends on `xhs`, `playwright`, `curl-cffi`, `requests`, and `lxml`; `XHS-Downloader` is checked out under `D:\Workshop\XHS-Downloader` as an external reference/tool and synced with its own uv environment, but it is not vendored or imported by `video-bundle-agent`. `MediaCrawler` is checked out under `D:\W\Codex\external\MediaCrawler` with its own uv environment and is invoked through a small adapter that runs its official `xhs detail` jsonl workflow.

The provider resolves Xiaohongshu URLs and `xhslink.com` short links, reads explicit Netscape cookies from `%APPDATA%\video-bundle-agent\xiaohongshu.cookies.txt` when provided, extracts note metadata/media from the web HTML initial state, writes `media.json`, downloads video/image media, transcribes downloaded videos with whisper.cpp, and runs visual recall against retained video files. Top-level comments are bounded to `--max-comments`, sorted by `like_count`, and use MediaCrawler's official detail/jsonl output; raw output is written under `raw/xiaohongshu/mediacrawler/`. The old bundled local signer, `xhs-signer`, `--xhs-sign-url`, and `XHS_SIGN_URL` path is removed because fresh-cookie API retries still produced `300011` account/session risk responses. MediaCrawler's saved CDP profile does not replace valid platform permissions: platform account/session risk responses such as `300011` are recorded as `PERMISSION_REQUIRED`; login-expired responses such as `-100` are recorded as `COOKIE_REQUIRED`.

MediaCrawler search, author-homepage traversal, nested comments, and bulk crawling remain deferred. The current MediaCrawler use is only for bounded top-level comment recovery and must not become bulk crawling or account-rotation behavior. Its normal saved-profile QR/SMS login flow is allowed when the user explicitly runs Xiaohongshu analysis; repeated login loops after a clear platform error are not.
