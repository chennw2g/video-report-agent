# Report Output Contract

This document is the canonical report-output contract for `video-report`.

Status: v1.1 content structure frozen on 2026-06-08. Visual baseline v1 is Template D / `Editorial Lab`,
documented in `docs/report-visual-style.md`. Future work should proceed to renderer implementation unless
the user explicitly reopens the content or visual contract.

## Skill Boundary

- `video-bundle-prep` prepares reusable evidence bundles. It chooses the provider workflow, classifies the
  video, chooses visual policy, extracts frames, checks readiness, and writes mode-independent
  `report.input.json`.
- `video-report` writes the final user-facing report from a prepared bundle or `report.input.json`.
- `video-report` may call the prep workflow automatically when the user gives a raw video link or local
  file, but evidence preparation and report writing remain separate responsibilities.
- The future AI plugin shell is deferred until quick/deep report structure and visual design are stable.

## Mode Rules

- Supported modes are `quick` and `deep`.
- Default mode is `quick`.
- Use `deep` only when the user explicitly asks for `深入分析`, `深度报告`, `详细解读`, or `deep 模式`.
- Do not auto-upgrade to `deep` because the video is dense, long, or tutorial-like.
- `quick` and `deep` use the same HTML/PNG design system. PDF is optional legacy/fallback export. The
  difference is content depth and evidence density, not a separate visual product.

## Reusable Evidence

- Bundle preparation is mode-independent.
- `report.input.json` is mode-independent and should stay a reusable evidence index.
- When `report.input.json.selected_evidence.visual_selection_plan.available` is true, use it as the primary
  pointer for which screenshots deserve inspection. The plan is agent-authored semantic guidance, not a
  replacement for reading the transcript or looking at the selected images.
- Respect `selected_evidence.selection.body_screenshot_policy`, especially `selective`: do not place body
  screenshots unless they materially clarify the adjacent analysis.
- The same prepared bundle should support `quick` first and `deep` later without recollecting source material.
- Existing bundle report runs should only perform light readiness/input checks unless the user explicitly
  asks to refresh source material.

## Output Files

- `quick`: `report.content.quick.json`, `report.zh.quick.html`, `report.zh.quick.png`
- `deep`: `report.content.deep.json`, `report.zh.deep.html`, `report.zh.deep.png`
- Default/compatibility aliases may point to the latest/default quick report:
  `report.content.json`, `report.zh.html`, `report.zh.png`
- PDF may still be produced as a secondary export when explicitly useful, but long PNG replaces PDF as the
  preferred shareable visual artifact. Long PNG uses the same HTML body without the left navigation.
- HTML/PNG reports keep a small provenance footer:
  `Generated by <agent> (Powered by <model>)`. Use `signature_agent` and `signature_model` in
  `report.content.<mode>.json` when the defaults need to be overridden. The agent is the execution
  agent/shell, such as `CODEX` or `Claude Code`, not the report project name.

## Quick Content

Goal: after reading the report, the user should basically know what the video says and decide whether it
is worth deeper study.

Default sections:

1. Basic information: title, platform, author/uploader, publish time, duration, link, and video type tags.
   Header metric cards must use this fixed order: `平台`, `作者`, `发布时间`, `视频时长`, `播放量`,
   `评论数`, `点赞数`, `分享数`. If a metric is unavailable, show `未获取`; do not substitute share
   count for like count or any other engagement metric.
2. AI multi-dimensional evaluation snapshot: render the compact rating graphic before the video overview
   so the reader sees AI's overall judgment immediately. Do not put detailed per-dimension reasoning here.
3. Video overview: concise description of what the video covers, including complete original chapters from
   `source_chapters.json` when present. Infer natural sections only when source chapters are unavailable.
   Current native chapter sources are Bilibili `view_points` and YouTube yt-dlp `chapters`; Xiaohongshu
   should usually use inferred natural sections.
   Keep this section mostly objective; put learning value, applicability,
   and judgment in AI critique. Do not force this into one sentence.
4. Core points: 3-7 important points. Embed timestamp screenshots beside the point they support only when
   the image materially clarifies the point.
5. AI critique: short text commentary explaining the most important rating judgments and source limits.
6. Audience feedback: 2-4 main feedback directions when comments are available; one-line missing/limited
   note when unavailable.
7. Attention notes and evidence files. Merge source cautions, diagnostics, and limitations into attention
   notes.

Image rules:

- `quick` is not required to include images in every section.
- The report hero visual should prefer an explicit `hero_visual`, then the platform cover/thumbnail
  (`metadata.thumbnail_path` when available, otherwise `metadata.thumbnail`), then a representative
  screenshot. Do not let the first inline body screenshot replace a valid platform cover by accident.
- Do not create a standalone key-image gallery. Put screenshots next to the text they clarify.
- Do not use screenshots as decoration or filler.
- For talking-head, podcast, interview, or other low-visual-variation videos, do not embed body screenshots
  by default. A frame that only repeats burned-in subtitles or shows the same speaker pose is not enough
  visual value; keep at most a representative hero image unless the frame adds independent information.
- Do not reuse the same screenshot in both the overview and a later core/chapter section. Keep it where it
  best supports the text.

## Deep Content

Goal: adapt the report to the video type. Deep is not just a longer quick report.

Default sections:

1. Basic information and source context. Header metric cards must use the same fixed order as `quick`:
   `平台`, `作者`, `发布时间`, `视频时长`, `播放量`, `评论数`, `点赞数`, `分享数`. Do not add a
   report-type card by default and do not replace a missing metric with another metric.
2. AI multi-dimensional evaluation snapshot: render the compact rating graphic before the video overview
   so the reader sees AI's overall judgment immediately. Keep detailed rating reasoning for the later
   AI critique section.
3. Video overview and content map.
4. Original chapter-by-chapter detail when `source_chapters.json` is present; otherwise natural-section detail.
5. Core viewpoint deep dive.
6. Audience feedback and visible disagreement, if comments are available.
7. AI-organized visuals, diagrams, timelines, matrices, or tables when they clarify the material.
8. AI detailed evaluation and critique using the same rating rules as `quick`.
9. Attention notes: merge source cautions, tool diagnostics, and report limitations near the end.
10. Evidence index.

Source-type adaptation:

- Tutorials, courses, software demos, and process-heavy videos should be reconstructed into a readable text
  tutorial with concepts, steps, prerequisites, pitfalls, examples, and important screen states.
- News, interviews, speeches, analysis, opinion, and market commentary should focus on claims, assumptions,
  evidence, context, implications, disagreements, and risks.
- Embed screenshots inside the relevant chapter/detail/core-viewpoint section.
- Do not place a screenshot in every chapter or viewpoint by default. Include an image only when it adds
  information that the text alone does not convey, such as an interface state, visual comparison, prompt
  table, timeline, chart, or screen-specific operation.
- For pure talking-head videos with little visual change, avoid inline body screenshots even in `deep`.
  Screenshots still remain bundle evidence and may be used as the hero representative image, but they should
  not be used as filler beside analysis paragraphs.
- Avoid repeated or near-identical screenshots in adjacent sections. When two screenshots are genuinely
  useful in the same block, render them compactly side by side when the viewport allows it.
- The same screenshot should not appear twice in the main report unless the user explicitly asks for an
  image appendix or comparison view.
- Do not add standalone `适用人群`, generic `学习笔记`, or generic `可执行清单` sections by default.
- Add checklists only when the source itself is procedural or the user explicitly asks for action items.

## AI Evaluation

- Both `quick` and `deep` include the same compact multi-dimensional rating graphic, such as a radar chart.
- The compact rating graphic should be the first content module after the top basic/source information,
  before the video overview.
- Use a 1-5 scale. Do not use percentages or 0-10 scores by default.
- Default dimensions:
  - 可信度
  - 原创性
  - 价值密度
  - 论证强度
  - 信息密度
  - 时效性
- Timeliness evaluates whether the video's information is stale, common, current, or meaningfully
  forward-looking. For time-sensitive subjects, verify freshness with lightweight web research before
  writing a strong timeliness judgment.
- The chart is a quick visual orientation aid and does not need per-dimension reasons beside it.
- Explain rating reasoning later in the AI critique/detailed evaluation section, not in the first
  rating snapshot.
- Label the chart as AI evaluation, not source content.

## Audience Feedback

- `quick` summarizes comments briefly and should not quote many raw comments.
- `deep` may analyze support, criticism, questions, supplemental information, and disputes.
- `deep` may cite short original comment text when it preserves useful audience voice.
- Every cited representative comment should include `like_count` when available.
- Include `reply_count` when it helps show disagreement or discussion intensity.
- Representative comments must be traceable to real items in `comments.json`. Do not invent placeholder
  comments. If a cited comment has missing or zero likes, do not present it as a high-like representative
  comment unless the source sample itself proves that value.
- Do not dump large raw comment lists.
- If the available comment sample is small or partial, say so explicitly and do not present it as the
  whole audience view.

## Evidence Attribution

- `quick` should use lightweight evidence attribution: timestamps, inline screenshots, comment `like_count`,
  and a short evidence-file list are enough.
- `deep` should use stronger attribution for key claims, disputed judgments, important data, and tutorial
  steps.
- Evidence forms include transcript timestamps, inline screenshots, short comment quotes with `like_count`,
  and the final evidence index.
- Do not cite every sentence. Important conclusions must be traceable back to bundle evidence.
- Keep the report readable; avoid turning normal prose into paper-style footnote clutter.

## Source vs AI Interpretation

- Distinguish source material from AI interpretation.
- Source material includes video overview, chapter details, source claims, transcript-backed points, and
  factual visual descriptions.
- AI interpretation includes evaluation, risk notes, viewpoint analysis, applicability judgments, and
  AI-organized tables, charts, or diagrams.
- Audience feedback and diagnostic/trust notes should also be visually or textually distinguishable from
  source content.
- Use lightweight labels when useful, such as `视频内容`, `AI 解读`, `观众反馈`, and `诊断提示`.
- Do not make every sentence carry a label. The goal is to prevent AI inference from being mistaken for
  the original author's claim.

## AI-Organized Visuals

- `quick` should only include the fixed AI multi-dimensional evaluation chart by default. Do not add
  extra AI-made charts unless the user asks.
- `deep` may include AI-organized tables, diagrams, flowcharts, comparison charts, matrices, or timelines
  when they clarify the video content.
- AI-organized visuals must be useful, not decorative.
- Tables in content maps and AI-organized modules should center each column by default for visual consistency.
- Do not force AI-organized visuals into tables. Choose the form that fits the content: process flow,
  timeline, matrix, comparison table, axis chart, or concise diagram.
- Label them as `AI 整理` or `AI 解读`.
- Do not present AI-organized visuals as original video screenshots or author-provided charts.
- When a AI-organized visual is based on the source, its basis should be traceable to transcript
  timestamps, sections, screenshots, or other bundle evidence.
- Good deep candidates include tutorial step flowcharts, concept maps, viewpoint comparison tables,
  risk/benefit matrices, and event timelines.

## Evidence Index

- Keep an evidence index at the end of both `quick` and `deep` reports.
- `quick` should keep it lightweight: key files such as `metadata.json`, `transcript.txt`,
  `transcript.segments.json`, `slides.json`, `comments.json`, `audience_feedback.json`, and
  `diagnostics.json` when present.
- `deep` may be more detailed: key screenshots used in the report, transcript/comparison files, cited
  comment files, and diagnostics that affect interpretation.
- The evidence index is for audit and follow-up inspection. Do not interrupt normal reading by overloading
  the body with file-path references.

## Attention Notes

- Both modes may include a lightweight `注意事项` module near the end.
- This module merges source cautions, diagnostic/trust notes, and report limitations. It is not a raw tool
  log.
- User-facing attention notes must be in Chinese. Do not expose raw diagnostic codes such as
  `MEDIA_DOWNLOAD_FAILED` as the visible title; translate or summarize them and keep raw codes traceable in
  `diagnostics.json` or the evidence index.
- Include only items that materially affect interpretation, such as:
  - automatic-subtitle vs transcription disagreement that affects key terms, names, numbers, or claims
  - missing or partial comments
  - screenshot/keyframe extraction failure or missing visual evidence
  - OCR absence when the video relies heavily on visible screen text
  - platform permission, cookie, CAPTCHA, risk-control, or rate-limit diagnostics that limit evidence
  - source claims, commercial assumptions, costs, tool availability, or timeliness points that need caution
- For blocking issues, report blockers and repair steps instead of writing a substantive content report.
- For non-blocking issues, keep the module concise.
- Do not maintain a separate limitations module by default; put report limits here.
